# TABtransporter 작업 기준

## 1. 제품 목표

TABtransporter는 기타와 베이스 기타 악보를 조옮김하고, 조옮김 후 실제 연주 가능한 TAB 운지를 다시 제안하는 도구다.

핵심은 단순히 음을 올리고 내리는 것이 아니다. 음높이를 바꾼 뒤, 기타나 베이스에서 어느 줄과 어느 프렛으로 치는 것이 현실적인지까지 보여줘야 한다.

## 2. 절대 조건

- 기타와 베이스 기타를 모두 지원한다.
- 모든 핵심 기능은 무료 라이브러리와 무료 도구만 사용한다.
- 유료 API, 유료 SDK, 유료 변환 서버에 의존하지 않는다.
- Guitar Pro 앱은 유료/상용 소프트웨어이므로 제품 의존성, 필수 설치 도구, 권장 작업 흐름에서 제외한다.
- 사용자가 실제로 넣을 파일은 PDF 또는 이미지라고 가정한다.
- `.gp3`, `.gp4`, `.gp5`, `.gpx`, `.gp` 같은 GP 파일 형식은 핵심 입력이 아니라 나중의 호환성 후보로만 둔다.
- 언어는 고정하지 않는다. 기능을 가장 잘 지원하는 무료 라이브러리가 있는 언어를 우선한다.
- 사용자의 악보 파일은 기본적으로 로컬에서 처리한다.
- 자동 결과를 정답처럼 숨기지 않는다. 애매한 운지는 반드시 사용자가 확인할 수 있게 표시한다.
- 요청받지 않은 범위의 파일이나 기능은 임의로 바꾸지 않는다.
- 매번 작업은 구현, 검증, commit, git push까지 끝내는 것을 완료 기준으로 한다.
- git repository, remote, 인증, 네트워크 문제로 push가 불가능하면 완료로 말하지 말고 정확한 사유를 남긴다.

## 3. 확정된 제품 결정

- 앱 형태는 데스크톱 앱이다. Windows에서 실행 가능한 `.exe` 배포를 1순위로 둔다.
- 입력은 PDF와 이미지 둘 다 MVP에 포함한다.
- OMR 인식 오류 수정은 가능한 범위를 모두 제공한다.
  - 음표 수정
  - 박자/마디 오류 수정
  - 쉼표/임시표 오류 수정
  - 코드명 오류 수정
  - TAB 숫자/줄/프렛 오류 수정
- MVP 악기는 6현 기타 표준 튜닝과 4현 베이스 표준 튜닝만 지원한다.
- 조옮김 방식은 키 선택, 반음 단위 이동, 카포 기준 변환을 모두 제공한다.
- 원곡 운지 유지 전용 모드는 MVP에서 제외한다.
- TAB 운지는 자동 추천을 기본으로 한다.
- 출력은 PDF와 이미지로만 제공한다.
- 저장 위치는 사용자가 선택할 수 있게 한다. 기본 저장 위치는 바탕화면이다.
- 재생 기능은 MVP에 포함한다.
- 화면 구조와 세부 디자인은 작업 도구 느낌을 기준으로 구현자가 결정한다.
- 개발 순서는 구현자가 실현 가능성과 위험도를 보고 결정한다.

## 4. 기술 선택

1차 구현 언어는 TypeScript로 간다.
앱 껍데기는 Electron을 1순위로 둔다.

이유:

- alphaTab이 TypeScript/JavaScript 환경에서 GP 계열 파일 형식, MusicXML, TAB 렌더링, 재생 기능을 지원한다.
- Electron은 Windows `.exe` 배포, 로컬 파일 접근, 로컬 OMR 프로세스 실행에 적합하다.
- Tonal.js 같은 무료 음악 이론 라이브러리를 함께 쓰기 좋다.
- React/Vite 기반 UI를 그대로 데스크톱 앱에 넣을 수 있다.

기본 스택:

- App: TypeScript
- Desktop shell: Electron
- UI: React
- Build: Vite 계열
- Packaging: Windows `.exe` 생성이 가능한 무료 패키징 도구
- 악보/TAB 로딩, 렌더링, 재생: alphaTab 우선 검토
- 음 이름, MIDI 번호, 조옮김 계산: Tonal.js 또는 자체 순수 함수
- PDF/이미지 악보 인식: 앱 안에서 바로 변환 흐름을 실행한다.
- PDF 변환은 사용자가 별도 프로그램을 수동으로 돌리는 방식이 아니라, 앱의 import 단계에서 처리한다.
- 무료 OMR 엔진은 Audiveris를 1순위 후보로 두되, UX에서는 "PDF 업로드 -> 자동 변환"으로 보이게 만든다.
- OMR 결과인 MusicXML을 내부 표준 입력으로 쓴다.
- GP3/GP4/GP5 파일 조작은 MVP 이후 호환성 기능으로 미룬다.

기준:

- alphaTab으로 가능한 것은 alphaTab을 먼저 쓴다.
- alphaTab이 편집/내보내기에서 부족한 부분은 내부 데이터 모델과 변환 로직으로 보완한다.
- 특정 기능에 더 강한 무료 라이브러리가 확인되면 언어 선택을 다시 검토한다.

## 5. 지원 입력 형식

MVP 우선순위:

1. PDF
2. 이미지
3. OMR 변환 결과 MusicXML
4. 앱 내부 예제 포맷

주의:

- 사용자는 기본적으로 PDF/이미지를 넣는다고 본다.
- 앱 내부 연산은 PDF/이미지를 바로 조옮김하지 않는다.
- 먼저 OMR로 MusicXML 또는 내부 악보 모델을 만든 뒤 조옮김한다.
- GP 파일 지원은 MVP 범위가 아니다.

PDF와 이미지는 최종 입력 UI에서는 받는다.
하지만 조옮김 엔진은 PDF/이미지 픽셀을 직접 다루지 않고, OMR 변환 후의 구조화된 악보 데이터를 다룬다.

PDF/이미지 흐름:

1. 사용자가 PDF 또는 이미지 업로드
2. 앱이 PDF/이미지를 무료 OMR 변환기로 바로 처리
3. MusicXML 또는 내부 악보 모델 생성
4. 원본과 변환 결과를 나란히 보여줌
5. 사용자가 인식 오류 확인
6. 조옮김 실행
7. 기타/베이스 TAB 운지 재배치
8. 최종 검수와 export

PDF 변환 UX:

- 사용자는 PDF를 넣으면 된다.
- 사용자가 Audiveris, MuseScore 같은 외부 앱을 먼저 직접 열 필요가 없어야 한다.
- 내부 구현은 무료 로컬 변환기, bundled helper, local process 중 라이선스상 안전한 방식을 선택한다.
- AGPL 같은 라이선스 영향은 반드시 확인한다.
- 라이선스 때문에 직접 번들이 위험하면 별도 local converter 프로세스로 분리한다.
- 그래도 사용자 경험은 앱 안에서 바로 PDF 변환으로 유지한다.

## 6. 출력과 저장

MVP 출력 형식:

- PDF
- 이미지

이미지 출력은 PNG를 1순위로 둔다.
JPG는 필요하면 추가한다.

저장 규칙:

- export 시 저장 위치를 사용자가 선택할 수 있어야 한다.
- 기본 저장 위치는 바탕화면이다.
- 앱 내부 임시 파일은 사용자에게 보이지 않는 앱 데이터 또는 임시 폴더에 둔다.
- 최종 산출물은 사용자가 고른 위치에 저장한다.

## 7. 악기 모델

기타와 베이스를 별도 하드코딩하지 않는다. 둘 다 `fretted instrument`로 다룬다.

악기 설정은 다음 값을 가진다.

- instrumentType: guitar 또는 bass
- stringCount
- tuning
- fretCount
- capo
- preferredPosition
- allowedFretRange

필수 프리셋:

- 6현 기타 표준 튜닝
- 4현 베이스 표준 튜닝

MVP 이후 후보:

- 5현 베이스 표준 튜닝
- 7현 기타
- 커스텀 튜닝

주의:

- 베이스는 기타보다 줄 수, 음역, 포지션 판단이 다르다.
- 베이스를 기타 TAB의 축소판처럼 처리하지 않는다.

## 8. 조옮김 로직

조옮김은 항상 두 단계로 나눈다.

1. Pitch transpose
   - 모든 음의 실제 pitch를 목표 키나 interval만큼 이동한다.

2. Fingering remap
   - 바뀐 pitch를 현재 악기 설정에서 가능한 string/fret 후보로 변환한다.
   - 후보 중 실제 연주성이 좋은 것을 점수화해서 제안한다.

후보 생성 규칙:

- 각 줄의 개방현 pitch를 기준으로 fret을 계산한다.
- 계산된 fret이 악기 설정의 fretCount 안에 있어야 한다.
- capo가 있으면 capo 위치를 반영한다.
- 같은 시점의 여러 음이 같은 줄을 동시에 쓰면 불가능 후보로 본다.

점수화 기준:

- 원래 줄과 가까운 후보를 우선한다.
- 이전 음과 포지션 이동이 적은 후보를 우선한다.
- 손가락 벌림이 지나치게 큰 후보는 낮게 본다.
- 개방현 느낌이 중요한 구간은 가능한 한 보존한다.
- 슬라이드, 해머링, 풀링, 벤딩 같은 주법은 가능한 한 유지한다.
- 빠른 구간에서 큰 포지션 점프가 생기면 경고한다.

중요:

- 자동 운지는 "가능한 운지 제안"이다.
- 최종 운지는 사용자가 듣고 직접 확인해야 한다.
- 앱은 확신이 낮은 마디를 표시해야 한다.
- MVP에서는 내부적으로 후보를 여러 개 만들 수 있지만, UI 기본값은 자동 추천 결과 1개다.
- 사용자는 인식 오류나 이상한 TAB 숫자를 직접 수정할 수 있어야 한다.

조옮김 입력 방식:

- 목표 키 선택
- 반음 단위 올림/내림
- 카포 위치 기준 변환

## 9. 카포 처리

카포는 조옮김 계산 옵션으로 제공한다.

지원:

- 카포 위치 선택
- 카포 적용 후 실제 sounding pitch 계산
- 카포 적용 상태에서 가능한 줄/프렛 재계산

제외:

- 원곡 손모양을 그대로 유지하는 전용 모드
- 카포만 올리고 운지 재배치를 생략하는 모드

베이스 카포는 MVP 기본 UX에서 숨긴다.

## 10. 재생 기능

재생 기능은 MVP에 포함한다.

목적:

- OMR 변환 결과 검수
- 조옮김 결과 검수
- TAB 운지 수정 후 확인

기본 기능:

- 재생/정지
- 마디 또는 구간 반복
- 템포 조절
- 현재 재생 위치 표시

## 11. UI와 디자인 기준

디자인은 음악 작업 도구처럼 보여야 한다.

피해야 할 것:

- AI 서비스 느낌의 과한 그라디언트
- 큰 랜딩 페이지
- 추상적인 장식 이미지
- 기능 설명을 길게 늘어놓는 화면
- 카드가 과하게 많은 마케팅형 레이아웃

권장 방향:

- 첫 화면부터 PDF/이미지 열기와 악보 작업 화면을 보여준다.
- 좌측은 파일/페이지/트랙/악기 설정으로 둔다.
- 중앙은 원본 PDF/이미지와 변환 악보/TAB 비교 화면으로 둔다.
- 우측은 조옮김/카포/운지 옵션과 인식 오류 목록으로 둔다.
- 하단은 재생 컨트롤과 현재 위치 표시로 둔다.
- 색은 절제된 중립 톤을 기본으로 하고, 경고와 선택 상태만 명확한 색을 쓴다.
- 버튼은 아이콘과 짧은 텍스트를 쓴다.
- 악보, TAB, 재생 컨트롤, 조옮김 옵션이 한 화면에서 자연스럽게 연결되어야 한다.
- "AI가 알아서 해줌" 같은 문구를 쓰지 않는다.

## 13. 테스트 기준

필수 테스트:

- PDF 샘플을 입력했을 때 OMR 변환 단계가 분리되어 동작하는지
- 이미지 샘플을 입력했을 때 OMR 변환 단계가 분리되어 동작하는지
- OMR 실패나 낮은 확신 결과를 사용자에게 숨기지 않는지
- 기타 6현 표준 튜닝에서 조옮김이 맞는지
- 베이스 4현 표준 튜닝에서 조옮김이 맞는지
- 같은 pitch가 여러 줄에서 나올 때 후보가 모두 생성되는지
- fretCount 밖의 음이 거부되는지
- 같은 박자에 같은 줄을 동시에 쓰는 후보가 거부되는지
- capo 적용 결과의 실제 sounding pitch가 맞는지
- 원곡 운지 유지 전용 모드가 노출되지 않는지
- 재생 위치가 악보/TAB 표시와 동기화되는지
- PDF export가 열리는 파일로 생성되는지
- 이미지 export가 열리는 파일로 생성되는지
- 저장 위치 기본값이 바탕화면인지
- 사용자가 저장 위치를 바꿀 수 있는지
- Windows `.exe` 빌드가 생성되는지

테스트 원칙:

- 음정 계산은 숫자로 검증한다.
- UI는 실제 화면에서 겹침과 잘림을 확인한다.
- 렌더링 라이브러리가 만든 결과도 최소 샘플로 직접 확인한다.
- PDF/이미지 인식 결과는 원본과 비교할 수 있어야 한다.

## 14. 외부 도구 기준

무료 외부 도구:

- Audiveris: PDF/이미지 악보를 MusicXML로 바꾸는 1순위 OMR 후보
- MuseScore Studio: 변환된 MusicXML 검수와 비교용
- TuxGuitar: GP/PTB/MIDI/TAB 파일을 무료로 열어 비교하고 검수하는 용도
- Power Tab Editor: PTB/GP 계열 파일을 무료로 열어 비교하는 보조 검수 후보

MVP에서는 PDF/이미지 입력이 핵심이므로 앱 안에서 PDF 변환이 바로 실행되어야 한다.
단, AGPL 라이선스 영향을 확인하기 전에는 Audiveris 코드를 앱에 직접 섞지 않는다.
우선은 무료 local helper 또는 별도 local converter 프로세스로 다루는 방향을 검토한다.
사용자에게는 앱 안에서 처리되는 PDF 변환 기능으로 제공한다.

## 15. 무료 도구 조사 결과

앱 내부 1순위 후보:

- alphaTab
  - 무료 MPL-2.0
  - TypeScript/JavaScript 중심
  - GP3-8, MusicXML, alphaTex 로딩
  - 표준보, 기타 TAB, 드럼 TAB 렌더링
  - 재생, 반복, transpose 기능 제공

- Tonal.js
  - 무료 MIT
  - TypeScript/JavaScript 음악 이론 라이브러리
  - note, interval, MIDI, transpose 계산에 적합

앱 내부 보조 후보:

- OpenSheetMusicDisplay
  - 무료 BSD-3-Clause
  - MusicXML 브라우저 렌더링에 강함
  - MusicXML 안의 기타 TAB 표시 가능
  - 편집기는 아니므로 렌더링 후보로만 둔다.

- VexFlow
  - 무료 MIT
  - 악보와 기타 TAB을 직접 그릴 수 있는 저수준 렌더링 라이브러리
  - MusicXML 파서는 아니므로 직접 모델을 그릴 때만 후보로 둔다.

- PyGuitarPro
  - 무료 LGPL-3.0
  - Python에서 GP3/GP4/GP5 읽기, 쓰기, 조작 가능
  - PDF/이미지 MVP에는 필요하지 않다.
  - GP 파일 호환성을 나중에 붙일 때만 후보로 둔다.

- music21
  - 무료 BSD-3-Clause
  - Python 음악 분석/변환 라이브러리
  - MusicXML과 transpose 처리에 강함
  - TAB 운지 전용은 아니므로 보조 엔진으로만 둔다.

앱 외부 무료 검수 도구:

- Audiveris
  - 무료, 오픈소스, AGPL-3.0
  - PDF/이미지 악보를 MusicXML로 변환하는 OMR 도구
  - 현재 입력 전략의 1순위 후보
  - 인식률은 완벽하지 않으므로 변환 후 검수 필요

- MuseScore Studio
  - 무료, 오픈소스, GPL-3.0
  - MusicXML import/export, PDF export, TAB 편집 가능

- TuxGuitar
  - 무료, 오픈소스, LGPL
  - 멀티트랙 기타 TAB 편집/재생
  - GP, Power Tab, TablEdit, MIDI 파일을 열 수 있음

- Power Tab Editor
  - 무료, 오픈소스, GPL-3.0
  - PTB와 GP 계열 파일 검수 후보
  - 핵심 앱 의존성은 아니고 비교 도구로만 둔다.

- LilyPond
  - 무료, GPL
  - 고품질 악보 engraving에 강함
  - 인터랙티브 TAB 편집용은 아니므로 MVP에서는 제외한다.

제외 대상:

- Guitar Pro 앱
  - 유료/상용 소프트웨어
  - 제품 의존성으로 쓰지 않는다.

- Flat.io, Soundslice 같은 웹 서비스
  - 무료 제한 또는 유료 기능이 섞임
  - 핵심 기능 의존성으로 쓰지 않는다.

## 16. 참고한 라이브러리와 근거

- alphaTab: GP3-8, MusicXML, 기타 TAB 렌더링, 재생, track transpose 기능을 제공하는 무료 MPL-2.0 라이브러리.
- Tonal.js: TypeScript/JavaScript 음악 이론 라이브러리. note, interval, transpose, MIDI 변환에 사용 가능하며 MIT 라이선스다.
- VexFlow: JavaScript 기반 오픈소스 악보와 기타 TAB 렌더링 API. alphaTab이 부족할 때 보조 렌더링 후보로 둔다.

## 17. 핵심 판단

개념적으로 조옮김은 pitch를 이동하는 문제다.

지금 우리가 만드는 앱의 진짜 문제는 이동된 pitch를 기타와 베이스에서 실제로 칠 수 있는 string/fret으로 다시 배치하는 것이다.

한 가지 직관:

음은 하나여도 손 위치는 여러 개다. 그래서 이 앱은 정답 하나를 찍는 도구가 아니라, 가능한 운지를 만들고 사람이 고르기 쉽게 해주는 도구로 만든다.
