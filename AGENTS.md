# TABtransporter 작업 기준

## 1. 제품 목표

TABtransporter는 기타와 베이스 기타 악보를 조옮김하고, 조옮김 후 실제 연주 가능한 TAB 운지를 다시 제안하는 도구다.

핵심은 단순히 음을 올리고 내리는 것이 아니다. 음높이를 바꾼 뒤, 기타나 베이스에서 어느 줄과 어느 프렛으로 치는 것이 현실적인지까지 보여줘야 한다.

## 2. 절대 조건

- 기타와 베이스 기타를 모두 지원한다.
- 모든 핵심 기능은 무료 라이브러리와 무료 도구만 사용한다.
- 유료 API, 유료 SDK, 유료 변환 서버에 의존하지 않는다.
- Guitar Pro 앱은 유료/상용 소프트웨어이므로 제품 의존성, 필수 설치 도구, 권장 작업 흐름에서 제외한다.
- 단, `.gp3`, `.gp4`, `.gp5`, `.gpx`, `.gp` 같은 GP 파일 형식은 무료 호환 라이브러리나 무료 호환 앱으로 읽을 수 있으면 지원 후보로 둔다.
- 언어는 고정하지 않는다. 기능을 가장 잘 지원하는 무료 라이브러리가 있는 언어를 우선한다.
- 사용자의 악보 파일은 기본적으로 로컬에서 처리한다.
- 자동 결과를 정답처럼 숨기지 않는다. 애매한 운지는 반드시 사용자가 확인할 수 있게 표시한다.
- 요청받지 않은 범위의 파일이나 기능은 임의로 바꾸지 않는다.

## 3. 기술 선택

1차 구현 언어는 TypeScript로 간다.

이유:

- alphaTab이 TypeScript/JavaScript 환경에서 GP 계열 파일 형식, MusicXML, TAB 렌더링, 재생 기능을 지원한다.
- 브라우저 기반 로컬 앱으로 만들면 무료 배포와 테스트가 쉽다.
- Tonal.js 같은 무료 음악 이론 라이브러리를 함께 쓰기 좋다.

기본 스택:

- App: TypeScript
- UI: React 또는 동급의 가벼운 TypeScript UI
- Build: Vite 계열
- 악보/TAB 로딩, 렌더링, 재생: alphaTab 우선 검토
- 음 이름, MIDI 번호, 조옮김 계산: Tonal.js 또는 자체 순수 함수
- GP3/GP4/GP5 파일을 직접 조작해야 하면 PyGuitarPro를 Python 보조 엔진 후보로 검토
- PDF/이미지 악보 인식: 앱 내부 MVP에서는 직접 구현하지 않는다. 무료 외부 도구인 Audiveris로 MusicXML 변환 후 가져오는 흐름을 기본으로 둔다.

기준:

- alphaTab으로 가능한 것은 alphaTab을 먼저 쓴다.
- alphaTab이 편집/내보내기에서 부족한 부분은 내부 데이터 모델과 변환 로직으로 보완한다.
- 특정 기능에 더 강한 무료 라이브러리가 확인되면 언어 선택을 다시 검토한다.

## 4. 지원 입력 형식

MVP 우선순위:

1. MusicXML
2. GP 계열 파일 형식
3. alphaTex 또는 앱 내부 예제 포맷

주의:

- 여기서 GP는 유료 Guitar Pro 앱이 아니라 파일 형식을 뜻한다.
- GP 파일 지원은 유료 앱 설치를 요구하지 않는 무료 호환 구현으로만 한다.

PDF와 이미지는 직접 악보로 이해하지 않는다.

PDF/이미지 흐름:

1. 사용자가 Audiveris로 MusicXML 변환
2. TABtransporter에서 MusicXML import
3. 원본과 비교하면서 오류 확인
4. 조옮김과 운지 재배치

## 5. 악기 모델

기타와 베이스를 별도 하드코딩하지 않는다. 둘 다 `fretted instrument`로 다룬다.

악기 설정은 다음 값을 가진다.

- instrumentType: guitar 또는 bass
- stringCount
- tuning
- fretCount
- capo
- preferredPosition
- allowedFretRange

필수 프리셋:

- 6현 기타 표준 튜닝
- 4현 베이스 표준 튜닝
- 5현 베이스 표준 튜닝
- 커스텀 튜닝

주의:

- 베이스는 기타보다 줄 수, 음역, 포지션 판단이 다르다.
- 베이스를 기타 TAB의 축소판처럼 처리하지 않는다.

## 6. 조옮김 로직

조옮김은 항상 두 단계로 나눈다.

1. Pitch transpose
   - 모든 음의 실제 pitch를 목표 키나 interval만큼 이동한다.

2. Fingering remap
   - 바뀐 pitch를 현재 악기 설정에서 가능한 string/fret 후보로 변환한다.
   - 후보 중 실제 연주성이 좋은 것을 점수화해서 제안한다.

후보 생성 규칙:

- 각 줄의 개방현 pitch를 기준으로 fret을 계산한다.
- 계산된 fret이 악기 설정의 fretCount 안에 있어야 한다.
- capo가 있으면 capo 위치를 반영한다.
- 같은 시점의 여러 음이 같은 줄을 동시에 쓰면 불가능 후보로 본다.

점수화 기준:

- 원래 줄과 가까운 후보를 우선한다.
- 이전 음과 포지션 이동이 적은 후보를 우선한다.
- 손가락 벌림이 지나치게 큰 후보는 낮게 본다.
- 개방현 느낌이 중요한 구간은 가능한 한 보존한다.
- 슬라이드, 해머링, 풀링, 벤딩 같은 주법은 가능한 한 유지한다.
- 빠른 구간에서 큰 포지션 점프가 생기면 경고한다.

중요:

- 자동 운지는 "가능한 운지 제안"이다.
- 최종 운지는 사용자가 듣고 직접 확인해야 한다.
- 앱은 확신이 낮은 마디를 표시해야 한다.

## 7. 카포 처리

기타는 조옮김 후 무조건 TAB을 새로 짜지 않는다.

다음 두 결과를 비교할 수 있어야 한다.

- 실제 음을 바꾼 새 TAB
- 원래 손모양을 유지하고 capo만 적용한 TAB

베이스도 기술적으로 capo 값을 가질 수 있지만, 기본 UX에서는 기타보다 낮은 우선순위로 둔다.

## 8. UI와 디자인 기준

디자인은 음악 작업 도구처럼 보여야 한다.

피해야 할 것:

- AI 서비스 느낌의 과한 그라디언트
- 큰 랜딩 페이지
- 추상적인 장식 이미지
- 기능 설명을 길게 늘어놓는 화면
- 카드가 과하게 많은 마케팅형 레이아웃

권장 방향:

- 첫 화면부터 파일 열기와 악보 작업 화면을 보여준다.
- 좌측은 파일/트랙/악기 설정, 중앙은 악보와 TAB, 우측은 조옮김/운지 옵션으로 둔다.
- 색은 절제된 중립 톤을 기본으로 하고, 경고와 선택 상태만 명확한 색을 쓴다.
- 버튼은 아이콘과 짧은 텍스트를 쓴다.
- 악보, TAB, 재생 컨트롤, 조옮김 옵션이 한 화면에서 자연스럽게 연결되어야 한다.
- "AI가 알아서 해줌" 같은 문구를 쓰지 않는다.

## 9. 작업 순서

1. 프로젝트 골격 생성
2. 무료 라이브러리 설치와 라이선스 확인
3. 악기 프리셋과 내부 데이터 모델 작성
4. MusicXML 또는 alphaTab 샘플 import 검증
5. pitch transpose 구현
6. guitar/bass fingering 후보 생성 구현
7. 후보 점수화와 경고 표시 구현
8. 악보/TAB 렌더링 연결
9. 재생과 구간 확인 기능 연결
10. 파일 export 가능 범위 검증

## 10. 테스트 기준

필수 테스트:

- 기타 6현 표준 튜닝에서 조옮김이 맞는지
- 베이스 4현 표준 튜닝에서 조옮김이 맞는지
- 베이스 5현에서 낮은 음역이 유지되는지
- 같은 pitch가 여러 줄에서 나올 때 후보가 모두 생성되는지
- fretCount 밖의 음이 거부되는지
- 같은 박자에 같은 줄을 동시에 쓰는 후보가 거부되는지
- capo 적용 결과와 실제 조옮김 결과가 구분되는지

테스트 원칙:

- 음정 계산은 숫자로 검증한다.
- UI는 실제 화면에서 겹침과 잘림을 확인한다.
- 렌더링 라이브러리가 만든 결과도 최소 샘플로 직접 확인한다.

## 11. 외부 도구 기준

무료 외부 도구:

- Audiveris: PDF/이미지 악보를 MusicXML로 바꾸는 용도
- MuseScore Studio: 변환된 MusicXML 검수와 비교용
- TuxGuitar: GP/PTB/MIDI/TAB 파일을 무료로 열어 비교하고 검수하는 용도
- Power Tab Editor: PTB/GP 계열 파일을 무료로 열어 비교하는 보조 검수 후보

이 도구들은 앱의 필수 런타임 의존성이 아니다. 사용자가 원본 PDF나 복잡한 TAB을 준비할 때 쓰는 보조 도구로 본다.

## 12. 무료 도구 조사 결과

앱 내부 1순위 후보:

- alphaTab
  - 무료 MPL-2.0
  - TypeScript/JavaScript 중심
  - GP3-8, MusicXML, alphaTex 로딩
  - 표준보, 기타 TAB, 드럼 TAB 렌더링
  - 재생, 반복, transpose 기능 제공

- Tonal.js
  - 무료 MIT
  - TypeScript/JavaScript 음악 이론 라이브러리
  - note, interval, MIDI, transpose 계산에 적합

앱 내부 보조 후보:

- OpenSheetMusicDisplay
  - 무료 BSD-3-Clause
  - MusicXML 브라우저 렌더링에 강함
  - MusicXML 안의 기타 TAB 표시 가능
  - 편집기는 아니므로 렌더링 후보로만 둔다.

- VexFlow
  - 무료 MIT
  - 악보와 기타 TAB을 직접 그릴 수 있는 저수준 렌더링 라이브러리
  - MusicXML 파서는 아니므로 직접 모델을 그릴 때만 후보로 둔다.

- PyGuitarPro
  - 무료 LGPL-3.0
  - Python에서 GP3/GP4/GP5 읽기, 쓰기, 조작 가능
  - GP 파일 변환이나 배치 처리에 필요할 때만 후보로 둔다.

- music21
  - 무료 BSD-3-Clause
  - Python 음악 분석/변환 라이브러리
  - MusicXML과 transpose 처리에 강함
  - TAB 운지 전용은 아니므로 보조 엔진으로만 둔다.

앱 외부 무료 검수 도구:

- MuseScore Studio
  - 무료, 오픈소스, GPL-3.0
  - MusicXML import/export, PDF export, TAB 편집 가능

- TuxGuitar
  - 무료, 오픈소스, LGPL
  - 멀티트랙 기타 TAB 편집/재생
  - GP, Power Tab, TablEdit, MIDI 파일을 열 수 있음

- Audiveris
  - 무료, 오픈소스, AGPL-3.0
  - PDF/이미지 악보를 MusicXML로 변환하는 OMR 도구
  - 인식률은 완벽하지 않으므로 변환 후 검수 필요

- Power Tab Editor
  - 무료, 오픈소스, GPL-3.0
  - PTB와 GP 계열 파일 검수 후보
  - 핵심 앱 의존성은 아니고 비교 도구로만 둔다.

- LilyPond
  - 무료, GPL
  - 고품질 악보 engraving에 강함
  - 인터랙티브 TAB 편집용은 아니므로 MVP에서는 제외한다.

제외 대상:

- Guitar Pro 앱
  - 유료/상용 소프트웨어
  - 제품 의존성으로 쓰지 않는다.

- Flat.io, Soundslice 같은 웹 서비스
  - 무료 제한 또는 유료 기능이 섞임
  - 핵심 기능 의존성으로 쓰지 않는다.

## 13. 참고한 라이브러리와 근거

- alphaTab: GP3-8, MusicXML, 기타 TAB 렌더링, 재생, track transpose 기능을 제공하는 무료 MPL-2.0 라이브러리.
- Tonal.js: TypeScript/JavaScript 음악 이론 라이브러리. note, interval, transpose, MIDI 변환에 사용 가능하며 MIT 라이선스다.
- VexFlow: JavaScript 기반 오픈소스 악보와 기타 TAB 렌더링 API. alphaTab이 부족할 때 보조 렌더링 후보로 둔다.

## 14. 핵심 판단

개념적으로 조옮김은 pitch를 이동하는 문제다.

지금 우리가 만드는 앱의 진짜 문제는 이동된 pitch를 기타와 베이스에서 실제로 칠 수 있는 string/fret으로 다시 배치하는 것이다.

한 가지 직관:

음은 하나여도 손 위치는 여러 개다. 그래서 이 앱은 정답 하나를 찍는 도구가 아니라, 가능한 운지를 만들고 사람이 고르기 쉽게 해주는 도구로 만든다.
